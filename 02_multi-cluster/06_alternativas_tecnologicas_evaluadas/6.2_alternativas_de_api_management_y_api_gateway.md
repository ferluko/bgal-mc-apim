# 6.2 Alternativas de API Management y API Gateway

Las evaluaciones distinguen dos necesidades tecnicas distintas: gobierno/exposicion de APIs externas (north-south) y transporte de alto volumen interno (east-west). Bajo esa premisa, se analizaron alternativas de API Management y API Gateway con foco en escalabilidad, costo y operabilidad en entorno regulado.

Esta evaluacion esta referenciada al proyecto APIM como caso modelador de la reingenieria OpenShift; las conclusiones se usan como patron base y se revalidan por dominio cuando se extienden al resto de la plataforma.

## Alternativas evaluadas

- **Kong Enterprise** (evaluado, PoC completada):
  - Fortalezas: madurez de producto, ecosistema amplio, integracion Kubernetes, enfoque declarativo.
  - Limitaciones: modelo economico desfavorable para trafico interno masivo y restricciones operativas en arquitectura evaluada.
  - Estado: no recomendado para el caso dominante east-west por costo/riesgo operativo; se mantiene como candidato para north-south sujeto a validacion tecnica/economica.

- **Tyk Enterprise** (evaluado, demo/validacion en curso):
  - Fortalezas: experiencia en open banking, enfoque multicluster y modelo de costos mas predecible.
  - Limitaciones: menor madurez regional, gaps de integracion y roadmap de capacidades clave en evolucion.
  - Estado: evaluado con cautela; sin cierre como opcion principal.

- **Red Hat Connectivity Link** (evaluado):
  - Fortalezas: integracion nativa con OpenShift, continuidad en ecosistema actual, apoyo a migracion.
  - Limitaciones: producto inmaduro para todos los requerimientos, alcance parcial para APIM completo y mayor acople a vendor.
  - Estado: considerado como alternativa de respaldo/continuidad, no como respuesta integral.

- **Solo.io / Gloo Gateway** (evaluado):
  - Fortalezas: base Envoy, soporte de estandares modernos, buena capacidad para dominio north-south.
  - Limitaciones: no resuelve por si solo todo el dominio east-west; depende de combinacion con otras capas.
  - Estado: candidato fuerte para gateway north-south; decision final abierta y en evaluacion comparativa con Kong.

- **Traefik Hub** (evaluado):
  - Fortalezas: soporte Gateway API y esquema de costo atractivo.
  - Limitaciones: observabilidad enterprise insuficiente y problemas de estabilidad/escalabilidad en pruebas de estres.
  - Estado: descartado.

- **Apigee / IBM API Connect** (evaluados):
  - Limitaciones relevantes para contexto actual: complejidad de adopcion, dependencia de nube/proveedor o bajo fit cloud-native para necesidades del banco.
  - Estado: descartados en etapas tempranas.

## Hallazgos transversales

- El trafico interno service-to-service representa el caso dominante (aprox. 80%, ~7.5B requests/mes), y el APIM tradicional introduce overhead no alineado a ese patron.
- El pricing por API call es inviable para East-West en este volumen; se requiere fixed pricing o esquemas equivalentes de costo predecible.
- El problema no es solo reemplazar producto: se requiere rediseño de arquitectura para eliminar hair-pinning y reducir latencia operacional.
- Se valida la separacion de dominios: capa APIM/API Gateway robusta para North-South (entre DMZ y workloads) y service mesh para East-West, evitando forzar un unico producto para ambos.
- La decision de gateway/APIM (Gloo/Kong) debe evaluarse en conjunto con red, identidad, seguridad y observabilidad para sostener operacion, compliance y coexistencia con RH 3Scale por fases.
