# 6.3 Alternativas de Service Mesh

La evaluacion de service mesh se centro en el dominio east-west multicluster, donde se concentran volumen y sensibilidad operacional. El objetivo fue resolver conectividad interna, seguridad y observabilidad con menor dependencia de patrones de red legacy.

La referencia primaria de esta evaluacion es el proyecto APIM como modelador de la reingenieria OpenShift, y su extrapolacion a otros dominios requiere revalidacion tecnica especifica.

## Alternativas evaluadas

- **Istio Ambient Mesh / Gloo Mesh (based on Ambient)**:
  - Fortalezas: enfoque sidecarless, menor overhead teorico que malla sidecar tradicional, buen encaje conceptual para multicluster.
  - Hallazgo critico de PoC: falla reproducible en trafico cross-cluster ante reciclado de endpoints (pod churn), con requests colgadas.
  - Estado: descartado para el dominio east-west de esta iniciativa por riesgo operativo.

- **Cilium Mesh (Isovalent Enterprise)**:
  - Fortalezas: conectividad multicluster L3/L4, observabilidad eBPF, rendimiento alto, enfoque open/estandarizable.
  - Limitacion: no reemplaza por si solo capacidades completas de API management L7 north-south.
  - Estado: seleccionado para dominio east-west multicluster.

- **Red Hat OpenShift Service Mesh (alternativa considerada)**:
  - Fortalezas: soporte enterprise y mayor madurez percibida frente a opciones emergentes.
  - Limitaciones: evaluacion sujeta a fit tecnico/operativo contra volumen, complejidad y requerimientos reales.
  - Estado: considerado en comparativa, sin cierre como seleccion principal en el tramo evaluado.

## Criterios de comparacion aplicados

- Estabilidad bajo fallas reales de endpoint/reconexion.
- Comportamiento multicluster en trafico de produccion simulada.
- Overhead operativo y complejidad de administracion.
- Integracion con seguridad (mTLS/politicas), observabilidad y gobierno.
- Portabilidad y riesgo de lock-in tecnico.

## Conclusiones de evaluacion

- El criterio decisivo no fue solo feature-set, sino comportamiento en escenarios de falla plausibles.
- La malla seleccionada para east-west debia demostrar resiliencia tecnica en PoC, no solo compatibilidad funcional.
- Se consolido un enfoque combinado: service mesh para east-west y gateway/APIM para north-south.
- La adopcion de mesh queda condicionada a madurez operacional de Kubernetes/CI-CD, para evitar incorporar complejidad sin capacidad real de operarla.
