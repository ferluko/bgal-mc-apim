# 6.4 Alternativas de networking y service discovery

Las alternativas de networking se evaluaron para eliminar hairpinning, reducir latencia y habilitar operacion multicluster con menor dependencia de cambios manuales en infraestructura compartida. El foco estuvo en automatizacion DNS, ruteo entre clusters y discovery de servicios con trazabilidad.

El analisis esta anclado en hallazgos del proyecto APIM tomado como caso modelador de la reingenieria OpenShift, y debe revalidarse al escalar a dominios con patrones de trafico distintos.

## Alternativas evaluadas

- **External DNS Operator + integracion con DNS corporativo**:
  - Propuesta para automatizar actualizaciones DNS y reducir cambios manuales en exposicion/failover.
  - Beneficio esperado: menor tiempo operativo y menor drift de configuracion.
  - Condicion: integracion coordinada con equipos/plataformas de red corporativa.

- **F5 GTM con automatizacion de configuraciones**:
  - Alternativa para enrutar trafico inter-sitio con health checks y conmutacion controlada.
  - Beneficio: continuidad con stack corporativo existente.
  - Limitacion: GTM no esta implementado ni planificado en el corto plazo; requiere alternativa transitoria de ruteo/failover.

- **CoreDNS para resolucion interna orientada a evitar hairpinning**:
  - Alternativa para resolver rutas internas sin salida innecesaria al borde.
  - Beneficio: reduccion de saltos y mejora de latencia en trafico interno.
  - Limitacion: requiere diseno coherente de naming/routing y gobierno operativo.

- **Cilium Cluster Mesh (candidato para conectividad L3/L4 y service discovery multicluster)**:
  - Aporta discovery y conectividad multicluster nativos con enfoque sidecarless para trafico east-west.
  - Beneficios esperados: menor cantidad de saltos de red, menor latencia y menor dependencia de flujos de hairpinning por balanceadores externos.
  - Complementa pero no sustituye capas de gobierno L7/API (north-south).

- **Integracion con Infoblox y ecosistema DNS corporativo**:
  - Considerada para estandarizar automatizacion de zonas/vistas en escenarios multicluster.
  - Limitacion: puede exigir inversiones de infraestructura y ajustes de proceso.

## Hallazgos de evaluacion

- La resolucion del problema no es un unico producto, sino una composicion de capacidades (mesh + DNS + balanceo + automatizacion).
- El estado actual activo-pasivo con switcheo por DNS presenta limites practicos (TTL/cache) para continuidad y recovery rapido.
- La evolucion objetivo es activo-activo donde la aplicacion lo soporte, con segmentacion por criticidad y capacidad HA.
- El nivel de automatizacion de red (DNS/WAF/LB) es determinante directo de resiliencia y tiempo de recuperacion.
- Las alternativas con menor friccion operativa y mejor encaje con governance corporativo fueron priorizadas.

## Cilium como candidato para beneficios objetivo

Con base en la documentacion tecnica, Cilium se mantiene como candidato fuerte para networking y service discovery. En terminos de consistencia de programa, en `6.3` ya figura como opcion seleccionada para el dominio east-west; en este capitulo se focaliza su evaluacion como parte de la composicion de capacidades de red end-to-end.

- Eliminar hairpinning en trafico interno, habilitando comunicacion directa service-to-service (pod-to-pod) para el dominio east-west.
- Mejorar performance de red en el caso dominante de trafico interno, reduciendo latencia y saltos innecesarios.
- Habilitar service discovery multicluster nativo y conectividad inter-cluster mas consistente para escenarios activo-activo.
- Mejorar observabilidad de red con eBPF/Hubble (flows, drops, DNS, dependencias) sin exigir instrumentacion de aplicacion.
- Reducir MTTR en incidentes de conectividad mediante trazabilidad de red mas precisa y troubleshooting orientado a evidencias.
- Reducir complejidad operativa frente a modelos sidecar tradicionales en cargas de alto volumen.
- Fortalecer seguridad de comunicaciones internas con politicas basadas en identidad de workloads y mTLS en la capa de malla.
- Mantener una base cloud-native alineada con estandares abiertos para reducir lock-in tecnico en la capa de red.

## Condiciones para capturar esos beneficios

- Mantener PoC con escenarios de falla reales (incluyendo pod churn cross-cluster) como criterio de pase.
- Validar integracion operativa con F5/DNS/WAF y gobierno de egress corporativo para evitar brechas entre capas.
- Confirmar modelo de soporte enterprise, runbooks y capacidades operativas del equipo para day 2.
- Operar Cilium como complemento de API gateway/APIM north-south, no como reemplazo de gobierno L7 externo.
