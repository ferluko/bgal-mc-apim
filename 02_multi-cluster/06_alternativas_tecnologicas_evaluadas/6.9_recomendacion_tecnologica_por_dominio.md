# 6.9 Recomendacion tecnologica por dominio

La recomendacion consolidada se basa en la evidencia de reuniones, evaluaciones comparativas y resultados de PoC ejecutadas. Se propone una seleccion por dominio para maximizar resiliencia, escalabilidad y control de riesgo, evitando forzar una unica tecnologia para todos los casos.

La recomendacion esta construida sobre el proyecto APIM como caso modelador de la reingenieria OpenShift; por eso funciona como baseline estrategico y debe aterrizarse con validaciones especificas en cada dominio no APIM.

## Recomendacion por dominio

- **API Gateway / APIM north-south (exposicion externa):**
  - Recomendacion: consolidar una capa APIM/API Gateway robusta entre DMZ y clusters de workload (incluyendo dominios en mesh), manteniendo Gloo Gateway y Kong Enterprise como candidatos principales.
  - Razon: fortalece gobierno L7 en north-south, permite coexistencia por fases/dominios y evita cierre prematuro de vendor antes de completar validaciones tecnicas y economicas.

- **Service mesh east-west multicluster (trafico interno):**
  - Recomendacion: Cilium Mesh (Isovalent Enterprise).
  - Razon: mejor comportamiento esperado en conectividad/performance multicluster y cierre de decision tras descarte de Ambient para patron critico.

- **Networking y service discovery:**
  - Recomendacion: combinacion de capacidades de mesh + automatizacion DNS corporativa por fases (External DNS y, cuando aplique, F5 GTM) + diseno de resolucion interna para evitar hairpinning.
  - Razon: reduce latencia, manualidad operativa y riesgo de desincronizacion entre sitios, permitiendo evolucion desde el estado actual hacia el objetivo multicluster.

- **Observabilidad:**
  - Recomendacion: enfoque hibrido con estandar OpenTelemetry, stack corporativo existente y capa eBPF para visibilidad de red/mesh; evaluar APM premium en servicios tier-1 segun costo-beneficio.
  - Razon: cobertura end-to-end progresiva sin ruptura operativa.

- **Gestion de secretos e identidad:**
  - Recomendacion: Vault como backend de secretos con patron declarativo en OpenShift (VSO/ESO segun dominio), y migracion gradual de API keys estaticas hacia OAuth2/JWT + mTLS.
  - Razon: mejora trazabilidad, revocacion, cumplimiento y seguridad por diseno.

- **Operacion multicluster y gobierno de flota:**
  - Recomendacion: modelo combinado ACM + GitOps distribuido (ArgoCD) + IaC (Terraform/Ansible), incorporando perfilado declarativo de clusters (cluster profiles as code) y evolucion controlada de alternativas cloud-native segun madurez.
  - Razon: aprovecha capacidades ya instaladas, habilita enforcement por perfiles y permite escalar gobierno sin frenar adopcion.

## Condiciones de implementacion

- Mantener criterio de decision basado en PoC con escenarios de falla reales, incluyendo pod churn cross-cluster.
- Preservar fixed pricing o mecanismos de costo predecible para dominio east-west.
- Asegurar configuracion declarativa, versionada y auditable como baseline transversal.
- Definir taxonomia y perfiles de cluster con guardrails minimos para day 0/1/2 antes de escalar la flota.
- Evitar lock-in no justificado mediante estandares abiertos y desacople de componentes.
- Cerrar seleccion final de API Gateway north-south (Gloo/Kong) con criterios integrales de seguridad, operacion, costo y soporte.

## Cierre

La recomendacion no es un reemplazo tecnologico lineal, sino una evolucion por dominios con foco en continuidad del negocio, reduccion de deuda tecnica y preparacion para escalamiento multicluster y nube en forma progresiva.
