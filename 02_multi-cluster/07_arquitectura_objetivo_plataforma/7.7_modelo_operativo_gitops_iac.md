# 7.7 Modelo operativo GitOps + IaC

El modelo operativo objetivo combina GitOps e Infrastructure as Code para estandarizar el ciclo de vida de clusters y aplicaciones en escala multicluster. Esta definicion reutiliza decisiones del proyecto APIM como modelador de practicas operativas para la reingenieria OpenShift.

La estrategia es separar claramente responsabilidades de plataforma, seguridad y delivery aplicativa, manteniendo trazabilidad completa de cambios en repositorios versionados.

## Componentes operativos

- **GitOps:** despliegue declarativo de configuraciones y aplicaciones con reconciliacion continua.
- **IaC:** provisionamiento de infraestructura base con herramientas tipo Terraform y automatizacion complementaria tipo Ansible.
- **Gobierno multicluster:** control central de politicas y ciclo de vida de clusters (por ejemplo, ACM en topologia hub-spoke).
- **Repositorio de verdad:** manifests y politicas gestionados por dominio, con pipelines de validacion.

Como patron de referencia, se considera una distribucion de ArgoCD por dominios funcionales (infra, seguridad/RBAC, aplicaciones y middleware/APIs) para escalar ownership sin perder gobierno.

## Ciclo day 0 / day 1 / day 2

- **Day 0:** creacion de cluster e infraestructura base.
- **Day 1:** configuracion inicial, seguridad baseline, conectividad y observabilidad minima.
- **Day 2:** operacion continua, upgrades, scaling, politicas y cambios de aplicacion.

## Beneficios esperados

- Menos dependencia de tareas manuales y conocimiento concentrado.
- Menor drift entre clusters y entornos.
- Rollback y auditoria nativos por versionado.
- Mayor velocidad de ejecucion sin perder control operativo.
