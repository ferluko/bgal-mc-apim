# 7.4 Arquitectura ingress/egress y DNS global

La arquitectura de ingress/egress y DNS global debe sostener operacion multicluster con enrutamiento predecible, failover rapido y minima intervencion manual. Este diseno se basa en aprendizajes del proyecto APIM y en las definiciones de red discutidas para la reingenieria OpenShift.

El objetivo es desacoplar la exposicion de servicios de la ubicacion fisica del cluster, habilitando continuidad de servicio con automatizacion de balanceo, DNS y verificacion de salud.

## Ingress objetivo

- Ingreso norte-sur por capa perimetral y componentes de seguridad de borde.
- Derivacion a gateways de API o ingress controllers segun tipo de trafico.
- Politicas de entrada estandarizadas por entorno y dominio.

## Egress objetivo

- Salida controlada por politicas de red y rutas definidas por dominio.
- Segregacion de egress sensible para integraciones externas.
- Trazabilidad de comunicaciones salientes para cumplimiento y auditoria.

## DNS global y failover

- Uso de DNS global con health checks para distribuir trafico entre clusters.
- Automatizacion de actualizaciones DNS (por ejemplo, con operadores tipo external-dns).
- Integracion con balanceo global para conmutacion entre sitios.
- Nomenclatura estandar para abstraer a los equipos de la topologia fisica.
- Integracion de capacidades GTM/F5 e infraestructura DNS corporativa (por ejemplo Infoblox) como parte del plano operativo.

## Resultado operativo esperado

- Menor dependencia de tickets manuales de networking.
- Menor tiempo de recuperacion ante caida de cluster o sitio.
- Escalado multicluster sin redisenar la exposicion publica en cada cambio.
