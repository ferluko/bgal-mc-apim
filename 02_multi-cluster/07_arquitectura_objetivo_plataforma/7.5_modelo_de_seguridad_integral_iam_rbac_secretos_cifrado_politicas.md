# 7.5 Modelo de seguridad integral

El modelo de seguridad objetivo combina identidad, cifrado, politicas y gobierno operativo para un entorno multicluster regulado. La referencia de implementacion parte del caso APIM, donde se identificaron debilidades del esquema actual y se definio una transicion hacia autenticacion moderna y seguridad por identidad de workload.

El principio rector es pasar de controles basados en credenciales estaticas y configuraciones manuales a un modelo declarativo, auditable y automatizable.

## Componentes del modelo

- **IAM y autenticacion:** integracion con identidad corporativa, OAuth2/JWT para consumo de APIs y RBAC granular por roles.
- **Service identity:** autorizacion entre servicios basada en identidad de Kubernetes y no en API keys estaticas.
- **Cifrado en transito:** mTLS para comunicaciones service-to-service y controles TLS en ingreso/egreso.
- **Secretos:** Vault como backend central con sincronizacion a Kubernetes mediante operadores (VSO/ESO segun dominio).
- **Cifrado y hardening de plataforma:** cifrado de ETCD y lineamientos de seguridad baseline por cluster.
- **Politicas de red:** aislamiento por defecto, reglas explicitas de comunicacion y control de egress.

El modelo explicita la migracion desde API keys estaticas (anti-patron identificado) hacia mecanismos con expiracion, revocacion y trazabilidad nativa.

## Enfoque operativo de seguridad

- Politicas de seguridad como codigo, versionadas y desplegadas por GitOps.
- Trazabilidad de cambios y evidencias para auditoria.
- Migracion gradual desde API keys a esquemas con expiracion y revocacion.
- Separacion de responsabilidades entre plataforma, seguridad y equipos de desarrollo.

## Resultado esperado

- Reduccion de riesgo por credenciales compartidas o permanentes.
- Mayor capacidad de control y cumplimiento regulatorio.
- Escalabilidad de seguridad consistente en todos los clusters.
