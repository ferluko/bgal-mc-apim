# 7.2 Patron norte-sur (APIs e ingress)

El patron norte-sur define la exposicion de servicios hacia consumidores externos al dominio interno de OpenShift, incluyendo internet, partners, canales y sistemas legacy/core. Esta definicion surge del frente APIM como caso modelador y se adopta como patron de referencia para la arquitectura objetivo de plataforma.

La arquitectura objetivo se organiza en tres capas: perimetro (DMZ y controles de borde), capa de API Gateway para gobierno L7 y capa de servicios internos. Esta separacion permite aplicar seguridad en profundidad, politicas de negocio y trazabilidad sin trasladar esa complejidad al trafico interno east-west.

## Flujo objetivo norte-sur

1. Entrada por DMZ con componentes perimetrales (F5/WAF/FW segun contexto).
2. Derivacion a API Gateway para autenticacion, autorizacion y politicas.
3. Enrutamiento controlado hacia servicios internos sobre la capa de conectividad de plataforma.

En este patron se considera tambien trafico norte-sur al proveniente de sistemas legacy y core bancario que consumen APIs alojadas en OpenShift.

## Capacidades esperadas en API Gateway

- OAuth2/JWT y mTLS para exposicion externa y B2B.
- Rate limiting, cuotas, transformaciones y versionado de APIs.
- Control plane para politicas y data planes distribuidos.
- Telemetria de transacciones, auditoria y observabilidad operacional.
- Integracion con IAM corporativo y portal de consumo cuando aplique.

## Criterios de gobierno

- Separacion clara entre gobierno de APIs externas y comunicacion interna entre servicios.
- Configuracion declarativa y versionada para evitar drift entre entornos.
- Despliegue gradual por dominios y coexistencia controlada durante migracion.
- Requisito de costos predecibles para no trasladar al dominio interno modelos de pricing no sostenibles.
