# 7.3 Patron este-oeste (service mesh)

El patron este-oeste define la comunicacion interna entre servicios y entre clusters. En la arquitectura objetivo, este dominio se resuelve con una malla de servicios sidecarless para reducir latencia, eliminar hairpinning y mejorar control de seguridad y observabilidad.

Como resultado de evaluaciones y PoC del frente APIM (caso modelador), se mantiene la decision de arquitectura sidecarless con Cilium Mesh para trafico east-west multicluster, descartando Ambient Mesh para este caso por riesgo operativo en un patron critico reproducible.

## Capacidades objetivo del patron este-oeste

- Service discovery nativo entre clusters.
- mTLS transparente entre workloads.
- Politicas de autorizacion por identidad de servicio.
- Routing y resiliencia de trafico entre dominios.
- Telemetria de red y dependencias a nivel servicio.
- Separacion operativa L4/L7: conectividad base centralizada y capacidades L7 avanzadas bajo demanda por namespace.

## Beneficios esperados

- Eliminacion del flujo de salida y reingreso por balanceadores externos para trafico interno.
- Reduccion de latencia y de complejidad operativa de networking.
- Menor dependencia de componentes legacy para autenticar trafico service-to-service.
- Mejor aislamiento de fallas y mejor capacidad de troubleshooting multicluster.

## Condiciones de adopcion

- Pruebas obligatorias de escenarios de pod churn cross-cluster.
- Integracion con observabilidad eBPF y trazabilidad end-to-end.
- Politicas declarativas gestionadas con GitOps para evitar divergencia entre clusters.
- Gateways east-west y politicas globales sincronizadas en todos los clusters productivos.
