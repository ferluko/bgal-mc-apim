# 3.2 Modelo operativo (dia 0, dia 1, dia 2)

## Dia 0

- El aprovisionamiento inicial de cluster requiere coordinación previa de infraestructura base (red, VIPs, DNS, almacenamiento y accesos).
- Durante este proceso, es necesario iniciar solicitudes vía tickets para la creación de recursos prerequisito que son gestionados por otros equipos, como por ejemplo redes, VIPs y zonas DNS.
- Se reporta necesidad operativa de 2-3 VIPs por cluster para API y tráfico de ingreso, con dependencias en equipos de red/comunicaciones.
- El despliegue se realiza sobre instalación IPI con customizaciones operativas del banco.
- Persisten pasos manuales y tickets para habilitar componentes de red y balanceo, lo que condiciona tiempos de provisión.
- La automatización existe de forma parcial en scripts/playbooks, pero no elimina la dependencia de aprobaciones y tareas manuales externas.

## Dia 1

- Aprovisionamiento orquestado por Ansible Automation Platform via RHACM
- El despliegue de un cluster base NO incluye integraciones de plataforma con F5, DNS corporativo y componentes de seguridad/identidad, los cuales requieren pasos manuales
- Se utilizan playbooks para acelerar alta de objetos de red (virtual servers, pools, monitores) pero no son orquestados, dado que el circuito sigue sujeto a coordinacion interequipos 
- Se establecen configuraciones iniciales de seguridad, RBAC y politicas base por entorno.
- La preparacion de componentes de exposicion de servicios (ingress, certificados, rutas) combina automatizacion tecnica con validaciones manuales.
- La estandarizacion de configuraciones entre sitios existe, pero con evidencias de sincronizaciones operativas no totalmente declarativas.

## Dia 2

- El proceso completo de operación Day 2 no está definido y la mayoría de las actividades se realizan de manera muy manual.
- La operación continua se estructura en tracks diferenciados: infraestructura/nodos, seguridad/RBAC y DevOps/release management.
- El modelo actual incorpora prácticas GitOps en algunos dominios, pero convive con procesos manuales para sincronización y gestión de cambios.
- La continuidad operativa en APIM mantiene pasos manuales relevantes (comparación de configuraciones, publicación y suscripción de APIs entre sitios).
- Se observan dependencias operativas persistentes en accesos privilegiados, tickets de red y procesos de certificados, que aumentan la carga de operación diaria.
