# 9.3 Framework tecnologico estandarizado para equipos

El framework estandarizado se define para reducir variabilidad tecnica y acelerar delivery en escala. La estrategia no es imponer una herramienta unica, sino consolidar un stack objetivo por capas que permita operar MC y APIM con seguridad, observabilidad y costo predecible.

## Capa runtime

- OpenShift/Kubernetes multicluster como sustrato comun.
- ArgoCD + GitOps para configuracion declarativa y reconciliacion continua.
- CloudBees como base actual de pipeline con evolucion a CD declarativo.
- Patrones de despliegue por dominio para escalabilidad operativa.

## Capa red y trafico

- Cilium Mesh Enterprise para este-oeste (sidecarless) en linea con APIM.
- Gloo Gateway (linea preferente) para norte-sur y gobierno de APIs externas.
- Gateway API como estandar para portabilidad y separacion de roles.
- DNS automatizado y rutas homologadas entre clusters y entornos.

## Capa seguridad

- IAM/RBAC estandarizado por dominio y criticidad.
- Vault + External Secrets para gestion centralizada de credenciales.
- Politicas de red y cifrado por defecto, con excepciones gobernadas.
- Trazabilidad de cambios y aprobaciones en repositorios y pipelines.

## Capa observabilidad

- OpenTelemetry como estandar de instrumentacion.
- Prometheus/Grafana para metricas operativas y troubleshooting.
- eBPF para visibilidad de red/dependencias a nivel kernel.
- Integracion progresiva con APM enterprise para servicios tier-1.

## Beneficios esperados

- Reduccion de friccion tecnica entre equipos y entornos.
- Menor tiempo de onboarding y mayor velocidad de entrega.
- Mayor resiliencia y menor costo de operacion por estandares comunes.
- Mejor posicion frente a lock-in por uso de patrones abiertos y portables.
