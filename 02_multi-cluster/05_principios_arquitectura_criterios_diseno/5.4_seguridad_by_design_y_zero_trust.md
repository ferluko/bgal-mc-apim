# 5.4 Seguridad by design y zero trust

Las transcripciones muestran que seguridad no puede seguir tratandose como capa posterior al despliegue. El principio objetivo es seguridad by design: identidad, acceso, secretos, cifrado y trazabilidad incorporados desde el diseno de plataforma y aplicados de forma consistente en todo el ciclo de vida.

El enfoque zero trust se define como "nunca confiar implicitamente, verificar siempre", incluso en trafico interno. Esto responde a brechas observadas en modelos legacy (credenciales estaticas, privilegios excesivos, controles heterogeneos) y al contexto regulado del banco.

- **Identity-first architecture:** integrar identidad corporativa como base de autorizacion para usuarios, servicios y automatizaciones.
- **Least privilege por defecto:** RBAC granular, eliminacion progresiva de cuentas compartidas y reduccion de privilegios permanentes.
- **Autenticacion moderna para servicios:** sustituir patrones legacy de API keys estaticas por esquemas robustos y revocables.
- **Secret management centralizado:** externalizar secretos fuera de codigo/repositorios, con rotacion y ciclo de vida controlado.
- **Cifrado end-to-end:** aplicar cifrado en reposo y en transito (incluyendo componentes de control y trafico servicio-a-servicio).
- **Microsegmentacion y politicas de red:** default deny, autorizacion explicita y control de comunicaciones por dominio/namespace.
- **Policy-as-code y enforcement continuo:** controles preventivos en pipelines y distribucion consistente de politicas entre clusters.
- **Auditabilidad nativa:** registrar y correlacionar accesos/cambios/eventos para cumplimiento, investigacion y respuesta a incidentes.

Como convergencia con Gartner G00841999 y `apim`, este principio tambien exige desacoplar seguridad de componentes propietarios puntuales, para mantener controles consistentes aun cuando evolucione el stack tecnologico.
